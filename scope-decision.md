## 1. 범위 조정 배경

### 원래 계획
- 7개 핵심 기능 (FR-1 ~ FR-7) 전체 구현
- 11개 Lambda 함수, AWS AI 서비스 7종 통합
- 10주 개발 일정 (주당 20시간)

### 현실
- 잔여 기간 4주, 15주차 시험기간 감안 시 실질 **3주**
- 사용 기술 **전부 미경험** (Jetpack Compose, CameraX, AWS AI 7종, Step Functions, SAM 등)
- 1인 개발, AI 에이전트 보조 활용하나 코드 이해·디버깅은 본인 몫
- 중간발표 12주차, 최종발표 15주차(시험기간)

### 판단
7개 기능을 얕게 구현하는 것보다, 핵심 3~4개를 깊이 있게 완성하여 **동작하는 데모 + 설명 가능한 코드**를 확보하는 것이 학습 프로젝트로서 더 가치 있다.

## 2. 구현 범위 분류

### 2.1 이번에 구현하는 기능 (v1.0 PoC)

| 우선순위 | 기능 | 관련 요구사항 | Lambda | AWS 서비스 |
|---|---|---|---|---|
| **필수** | 인프라 + 인증 | TR-INF1~11, NFR-S3 | fn-health | Cognito, API Gateway, S3, DynamoDB |
| **1순위** | FR-1 장소 인식 | FR-1.1~1.4, UC-1 | fn-place | Rekognition, Location Service, Bedrock, Polly |
| **2순위** | FR-3 메뉴판 OCR | FR-3.1~3.4, UC-3 | fn-menu | Textract, Translate, Bedrock |
| **3순위** | FR-7 채팅 일정 | FR-7.1~7.3, UC-7 | fn-chat | Location Service, Bedrock |

---

#### 필수: 인프라 + 인증

**포함 범위**
- AWS 계정, IAM 역할, S3 버킷 3종, DynamoDB 테이블, API Gateway REST API
- Cognito User Pool + Amplify SDK 연동
- Android 앱 스켈레톤 (Kotlin + Compose, 네비게이션, 인증 화면)
- SAM 템플릿 초기 구성

**선정 이유**
- 모든 기능의 전제 조건. 이것 없이는 어떤 기능도 동작하지 않음
- 인프라 구축 자체가 AWS 서버리스 아키텍처 학습 (L-2, L-4)

---

#### 1순위: FR-1 장소 인식

**포함 범위**
- 카메라 촬영 → S3 업로드 → Rekognition 분석 → Location 역지오코딩 → Bedrock 설명 생성
- 결과 카드 UI + Polly 음성 재생
- DynamoDB Place 테이블 저장 + 히스토리 조회

**선정 이유**

| 근거 | 설명 |
|---|---|
| **데모 임팩트 최대** | "카메라로 건물을 찍으면 AI가 설명해준다" — 한 문장으로 프로젝트의 핵심 가치를 보여줌. 중간발표 데모로 가장 강력함 |
| **아키텍처 대표성** | 카메라 입력 → S3 → Lambda → AI 서비스 3종 → DynamoDB → 앱 표시. 이 하나가 전체 아키텍처의 축소판으로, 시스템 설계 역량을 증명 |
| **AWS AI 서비스 3종 통합** | Rekognition + Location Service + Bedrock. 단일 기능으로 AI 서비스 통합 패턴을 가장 많이 학습 가능 |
| **2단계 파이프라인 검증** | ADR-005에서 결정한 "특화 서비스 1차 추출 → Bedrock 2차 생성" 패턴을 실전 검증 |

---

#### 2순위: FR-3 메뉴판 OCR

**포함 범위**
- 메뉴판 촬영 → Textract 텍스트 추출 → Translate 번역 → Bedrock 메뉴 추천
- 촬영 가이드라인 오버레이 UI
- 원문/번역/추천 카드 UI
- 식이 제한 프로필 설정

**선정 이유**

| 근거 | 설명 |
|---|---|
| **장소 인식과 파이프라인 구조 유사** | 사진 → S3 → Lambda → AI 서비스 → 결과 표시. FR-1에서 구축한 앱↔백엔드 파이프라인을 그대로 재활용하므로 추가 공수가 상대적으로 적음 |
| **새로운 AI 서비스 2종 추가** | Textract + Translate. FR-1의 3종과 합치면 AI 서비스 5종 통합 달성 |
| **여행 UX의 핵심 페인 포인트** | "외국어 메뉴판을 못 읽겠다"는 여행자의 가장 빈번한 불편. 실용성과 공감을 동시에 전달 |
| **Bedrock 프롬프트 심화** | 단순 설명 생성(FR-1)에서 사용자 프로필(알레르기) 기반 개인화 추천으로 프롬프트 활용도를 한 단계 높임 |

---

#### 3순위: FR-7 채팅 일정

**포함 범위**
- 텍스트 채팅 UI
- Location Service POI 검색 + Bedrock 대화형 추천
- 추천 결과를 일정에 추가
- 대화 컨텍스트(이전 이력) 유지

**축소 사항**
- 날씨 연동(FR-6) 제외 — 순수 위치 기반 추천만
- EventBridge 일정 알림 제외 — 일정 저장까지만

**선정 이유**

| 근거 | 설명 |
|---|---|
| **"AI 동행자" 정체성의 핵심** | 프로젝트명 Polylog의 핵심 가치는 AI와의 대화. 채팅 기능이 없으면 "AI 동행자"가 아니라 "AI 카메라 앱"으로 전락 |
| **Bedrock 활용의 깊이** | FR-1(단발 생성), FR-3(개인화 생성)에 이어 FR-7(대화형 컨텍스트 유지)로 Bedrock 활용 수준을 3단계로 보여줌 |
| **추가 AI 서비스 없음** | Location Service + Bedrock — 이미 FR-1에서 사용한 서비스만으로 구현 가능. 새로운 서비스 학습 부담 없음 |
| **발표에서 라이브 데모 가능** | 실시간으로 "근처 카페 추천해줘"를 입력하고 결과를 보여주는 인터랙티브 데모가 가능하여 발표 임팩트 높음 |

#### FR-2 실시간 통역 — 이후 구현 사유

| 항목 | 내용 |
|---|---|
| **핵심 사유** | **Transcribe Streaming이 WebSocket 기반으로, 프로젝트 내 다른 모든 API(REST)와 프로토콜이 다름.** 오디오 인코딩 형식, 청크 사이즈, VAD 설정, WebSocket 연결 관리 등 구현 난이도가 단독으로 가장 높음 |
| **위험도** | RISK-API2 (위험 등급: 상). 이 기능 하나에 1주 이상 소요될 가능성 높음 |
| **대안** | 텍스트 입력 → Translate → Polly 흐름의 **간소화 번역 기능**을 FR-7 채팅 탭 내에 부분 통합 가능. Transcribe Streaming은 v2.0에서 도전 |
| **학습 가치** | 높지만, 실패 시 다른 기능 일정까지 잠식. 현재 3주로는 리스크가 보상보다 큼 |

---

#### FR-4 자동 여행기 — 이후 구현 사유

| 항목 | 내용 |
|---|---|
| **핵심 사유** | **Step Functions(ASL)의 학습 곡선이 프로젝트 내 최고.** Map State 병렬 처리, Catch/Retry 에러 핸들링, Lambda 4종 오케스트레이션 — 이 기능 하나가 사실상 미니 프로젝트 규모 |
| **위험도** | RISK-API8 (위험 등급: 상). Lambda 4종(orchestrator, collector, photo-analyzer, writer) 각각을 개발하고 통합해야 함 |
| **대안** | 발표 시 **"아키텍처 설계는 완료했으나 구현은 v2.0 범위"**로 제시. ADR-007에 Step Functions 선정 근거와 대안 비교가 이미 문서화되어 있어, 설계 역량은 충분히 어필 가능 |
| **학습 가치** | 매우 높지만, 3주 안에 다른 기능과 함께 소화하기엔 병목이 됨 |

---

#### FR-5 영수증 가계부 — 이후 구현 사유

| 항목 | 내용 |
|---|---|
| **핵심 사유** | **프로젝트 핵심 가치("AI 여행 동행자")와의 연관성이 상대적으로 낮음.** 장소 인식·메뉴판·채팅은 "여행 중 실시간 도움"이라는 핵심 가치를 직접 전달하지만, 가계부는 "여행 후 정리" 성격이 강함 |
| **위험도** | RISK-API4 (위험 등급: 중). Textract AnalyzeExpense 응답의 Block 계층 구조 파싱이 복잡 |
| **대안** | 메뉴판 OCR(FR-3)에서 Textract를 이미 사용하므로, v2.0에서 AnalyzeExpense로 확장하는 것은 공수가 크게 줄어듦. 기반이 갖춰진 상태에서 추가하기 좋은 기능 |
| **학습 가치** | Textract의 다른 API(AnalyzeExpense)를 추가 학습하는 것으로, FR-3에서 기본 학습이 선행된 후에 하는 것이 효율적 |

---

#### FR-6 날씨 기반 일정 — 이후 구현 사유

| 항목 | 내용 |
|---|---|
| **핵심 사유** | **단독으로는 가치가 제한적이고, FR-7(채팅 일정)과 결합해야 의미가 있음.** 날씨 조회 자체는 외부 API 호출로 단순하지만, "날씨 기반 일정 재추천"이라는 완전한 UX를 구현하려면 FR-7의 채팅 컨텍스트 + 기존 일정 수정 로직이 성숙해야 함 |
| **위험도** | 낮음 (외부 REST API 단순 호출) |
| **대안** | FR-7 채팅 일정의 v1.0을 먼저 완성 후, v2.0에서 날씨 정보를 컨텍스트로 추가 주입하는 것이 자연스러운 확장 경로 |
| **학습 가치** | 외부 API 호출 + 캐싱 패턴 학습이지만, AWS AI 서비스 학습이라는 핵심 목표와는 거리 있음 |

## 5. 발표 전략

### 중간발표 (12주차)
- **데모**: FR-1 장소 인식 E2E — "카메라로 찍으면 AI가 설명해줍니다"
- **어필 포인트**: 서버리스 아키텍처 설계, AWS AI 서비스 3종 통합 패턴, ADR 기반 의사결정 과정

### 최종발표 (15주차)
- **데모**: FR-1 + FR-3 + FR-7 — "찍고, 읽고, 물어본다"의 3가지 상호작용
- **어필 포인트**:
  - 구현된 3개 기능으로 2단계 파이프라인 패턴(ADR-005) 실증
  - Bedrock 활용의 3단계 깊이 (단발 생성 → 개인화 추천 → 대화형 컨텍스트)
  - 미구현 기능은 **확장 로드맵**으로 제시 — "설계는 완료, 구현은 v2.0"
  - 기획 산출물 (planning, requirements, WBS, ADR, scope-decision)로 기획력 어필

### 예상 질의응답 대비

| 예상 질문 | 대비 답변 |
|---|---|
| "왜 7개 다 안 했나요?" | "3주 안에 이해 없이 AI로 코드만 생성하는 것보다, 3개를 깊이 이해하고 설명할 수 있는 것이 학습 프로젝트의 본질이라 판단했습니다. 범위 조정 근거는 scope-decision.md에 문서화되어 있습니다." |
| "나머지는 어떻게 할 건가요?" | "아키텍처 설계와 ADR은 7개 기능 전체에 대해 완료되어 있고, v1.0에서 구축한 파이프라인을 재활용하면 v2.0 확장 공수가 크게 줄어듭니다." |
| "통역 기능은 왜 뺐나요?" | "Transcribe Streaming이 WebSocket 기반으로 다른 REST API들과 프로토콜이 달라 구현 난이도가 독립적으로 높았고, 실패 시 다른 기능 일정까지 잠식할 위험이 있어 v2.0으로 분리했습니다." |
