## [4] 아키텍처·적용기술·성능 — 1:10 ⏱2:40  (그룹 B ★ 최고 효율)
【4p 4기능 파이프라인 → 5p 전체 아키텍처】

> "네 기능 모두 **같은 한 줄짜리 흐름**을 탑니다.
> 앱 입력 → **API Gateway**(요청 라우팅) → **Lambda**(기능별 로직) → **AWS AI·외부 API** → **DynamoDB·S3**(저장) → 결과.
>
> 기술 선택의 이유를 흐름 따라 말씀드리면 —
> • 앱은 **Flutter** 한 코드베이스로, 3주 PoC에 가장 효율적입니다(ADR-003, Android 타깃).
> • 서버는 **API Gateway + Lambda 7종**, **상시 서버 0대**·요청당 과금이라 **무료티어 월 2~3달러**로 운영됩니다(ADR-001).
> • 인증은 **Google 소셜 로그인**, ID 토큰을 `fn-authorizer`가 **Google tokeninfo로 검증**하는 **무상태 인가**입니다(ADR-007 — Cognito 우회).
> • AI는 **Bedrock의 Claude로 일원화**했습니다 — 장소 추천·대화형 일정, 그리고 **영수증 글자 인식까지 Bedrock 비전 한 번의 호출**로 처리합니다(InvokeModel). 메뉴 번역만은 앱이 직접 하지 않고 **구글 렌즈에 위임**합니다(아래 [5]에서 그 이유 설명).
> • 저장은 **DynamoDB**, `trip_id` 파티션 + **시각을 정렬키(SK)**로 둬서 일정·지출이 **자동 시간순 정렬**됩니다(ADR-014/015). 사진 원본은 **S3 + Presigned URL**(SSE 암호화, ADR-008).
>
> 성능·비용 최적화는 세 가지입니다.
> **Haiku 모델 + 프롬프트 토큰 최소화로 지연·비용↓**(ADR-004/009), **정렬은 키 설계로 비용 0**, **서버 0대로 고정비 0**.
>
> 구조는 — 앱은 **feature 단위 디렉토리**(`recommend·menu·receipt·schedule`), 백엔드는 **기능별 Lambda 분리**로 장애 영향 범위를 격리했습니다."

> **📊 채점 포인트 (★최고효율 구간 — Cat1·Cat2·Cat3 동시 득점)**
> - ☑ **(Cat1) 적용 기술 표현**: Flutter · API Gateway · Lambda 7종 · Bedrock(Claude) · DynamoDB · S3 — 스택을 **빠짐없이**
> - ☑ **(Cat1) 어떻게 구현했는지**: "앱 입력 → API Gateway → Lambda → AI/외부API → DynamoDB·S3 → 결과" **한 줄 파이프라인**으로 설명
> - ☑ **(Cat3) 적용기술 설명을 읽지 않고**: 기술 선택 이유를 **흐름 따라 암기**로 (ADR 번호까지)
> - ☑ **(Cat3) 아키텍처 이해 설명**: 5p 다이어그램을 보며 **왜 이 구조인지** (무상태 인가·키 정렬 등)
> - ☑ **(Cat3) 성능 최적화 노력**: **3가지 명시** — ① Haiku+프롬프트 토큰 최소화(지연·비용↓) ② 키 설계로 정렬 비용 0 ③ 서버 0대로 고정비 0
> - ☑ **(Cat2 질의응답) 앱 구조**: **레이어·디렉토리 구조** = feature 단위 디렉토리 + 기능별 Lambda 분리
> ⚠️ 빠지면 감점: 성능 3가지 중 누락 / "구글 렌즈 위임" 언급 빠지면 메뉴=Bedrock으로 오답([5]·부록A와 모순)

## [5] ADR 시행착오 — 0:50 ⏱3:30  (그룹 D ★)
【6p ADR before→after 표】

> "이 과정은 매끄럽지 않았고, **결정을 네 번 번복**했습니다. 전부 ADR에 기록돼 있습니다.
>
> ① **Cognito → 소셜 OAuth**(ADR-007): 공용 계정 IAM에서 Cognito가 막혀, ID 토큰 직접 검증으로 우회했습니다. *제약을 만나 설계를 바꾼 사례.*
> ② **메뉴 번역을 직접 만들었다가 → 「구글 렌즈」 위임으로 전환**: 처음엔 사진을 Bedrock 비전으로 읽어 번역·메뉴 추천까지 앱이 다 했지만, **작은 비전 모델 한 호출에 읽기·번역·추천을 몰아넣자 품질이 들쭉날쭉**했고, 환경 제약상 더 좋은 도구를 붙이기 어려웠습니다. 실시간 카메라 번역에 특화된 **구글 렌즈로 위임**하니 모든 언어가 안정적으로 처리됐습니다. *직접 만드는 것보다 검증된 도구를 쓰는 게 사용자에겐 더 나은 판단.* (영수증은 정형 데이터라 Bedrock 비전 유지)
> ③ **CloudFront 철회**(ADR-008): PoC 트래픽엔 CDN이 과투자고 영수증은 개인정보라 Presigned URL로 충분했습니다. *오버엔지니어링 회피.*
> ④ **대화형 플래너를 별도 Lambda로 분리**(ADR-017): 30초 걸리는 무거운 플래너가 가벼운 일정 CRUD의 타임아웃까지 끌어내려, `fn-planner`를 떼어내 영향 범위를 격리했습니다.
>
> **ADR 17건으로 모든 결정의 근거·대안·번복을 기록**했습니다."

> **📊 채점 포인트 (★ Cat3 시행착오 + Cat2 질의응답 준비)**
> - ☑ **(Cat3) 구현 시행착오를 개발자 관점에서**: "결정을 네 번 번복" — 각 번복마다 **개발자가 만난 문제→판단** 구조
> - ☑ **(Cat2) ADR 최소 3개 표시**: ADR-007·008·017 등 **4건 번호와 함께** 제시 (기준은 "최소 3개")
> - ☑ **(Cat2) ADR 기록 기준으로 정확하게(훈련되게)**: before→after 표(6p)로 **근거·대안·번복**을 일관되게
> - 번복별 핵심 키워드(반드시 이유까지): ① Cognito→소셜OAuth = *제약 회피* / ② 메뉴 직접→구글렌즈 = *검증된 도구가 더 나음* / ③ CloudFront 철회 = *오버엔지니어링 회피* / ④ 플래너 Lambda 분리 = *영향범위 격리*
> ⚠️ 빠지면 감점: ADR **번호**를 안 부르면 "기록 기준 정확" 점수 약화 / 번복 "이유"를 생략하면 단순 변경으로 보임

## 부록 A. 질의응답(Cat2) 백업 — 답변 3종 암기
1. **왜 Cognito가 아닌 소셜 OAuth?** 공용 계정 IAM에서 Cognito가 차단 → Google ID 토큰을 `fn-authorizer`가 **tokeninfo로 검증**하는 무상태 방식으로 우회(ADR-007).
2. **Bedrock 리전?** 인프라는 서울, **Bedrock만 us-east-1 cross-region**(~200ms). 프롬프트 최적화로 상쇄(ADR-009).
3. **왜 DynamoDB?** 여행 단위 단순 액세스 + On-Demand 과금이 PoC에 적합. **`trip_id` + 시각 SK** 설계로 정렬 비용 0(ADR-002/014/015).
- (보너스) **메뉴 번역은 왜 직접 안 하고 구글 렌즈인가?** 작은 비전 모델 한 호출에 읽기·번역·추천을 몰면 품질이 불안정 + 환경 제약 → 실시간 카메라 번역에 특화된 렌즈로 위임(`menu_screen.dart`). 영수증은 정형 데이터라 Bedrock 비전 유지.
- (보너스) **월 비용?** 무료티어 ~$2~3, 종료 후 $10~15.
